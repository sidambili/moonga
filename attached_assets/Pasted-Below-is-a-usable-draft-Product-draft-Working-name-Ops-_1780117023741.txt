Product draft

Working name

Oncident

Purpose

Create one control plane that connects GitHub, Linear, Better Stack, Sentry, Slack, and email.
It receives operational events, opens an AI agent session, gathers context, drafts analysis, and routes human-approved actions.

Core principle

The system does not auto-fix code in v1.
It triages, drafts, explains, and routes.

⸻

What the product does

Trigger	Agent session output	Human action
Error from Sentry or Better Stack	probable cause, affected service, likely file area, Slack summary, Linear draft	approve or edit in Slack
New Linear ticket	implementation plan, scope, dependencies, suggested files, acceptance criteria	approve or edit in Slack
Deployment anomaly	incident summary, severity, service impact, next steps	approve or escalate
GitHub event	change summary, risk notes, linked ticket context	review in Slack

Build scope

v1 must do

Capability	Description
Webhook ingestion	Receive events from GitHub, Linear, Better Stack, Sentry, Slack
Event normalization	Convert all inbound data into one internal schema
Session creation	Create one agent session per event or ticket

Agent reasoning	Produce diagnosis, plan, and summary
Slack approval	Send draft to Slack with approve, edit, reject

Email integration	Send notifications or approval summaries
Model routing	Choose model per task, with BYOK support

v1 should not do

Capability	Reason
Auto code changes	Too risky too early
Auto deploys	Too much blast radius
Multi-agent choreography	Unnecessary for first version
Deep workflow branching	Adds complexity before product value is proven
Full PR automation	Phase 2 only

Internal object model

1. Event

Represents the raw trigger.

Field	Purpose
event_id	unique identifier
source	GitHub, Linear, Better Stack, Sentry, Slack, email
event_type	error, anomaly, ticket_created, issue_updated, deploy_failed
severity	low, medium, high, critical
service	mapped service name
repo_id	linked repo
ticket_id	linked Linear ticket
payload_raw	original webhook data
created_at	timestamp

2. Agent session

Represents the AI work unit.

Field	Purpose
session_id	unique identifier
event_id	parent trigger
objective	diagnose, plan, summarize, draft
status	pending, running, needs_review, approved, rejected, completed
model_used	selected provider/model
context_snapshot	collected data
output_summary	final AI result

3. Artifact

Represents outputs the human can review.

Field	Purpose
artifact_id	unique identifier
session_id	parent session
type	slack_message, linear_ticket, incident_report, implementation_plan
content	generated text
approval_state	draft, approved, rejected, edited

Agent behavior

Incident session

When an alert arrives, the agent should:

1. Identify the service and repo.
2. Pull recent logs and recent commits.
3. Check for related Linear tickets.
4. Read relevant source files.
5. Draft:
    * what happened
    * likely root cause
    * severity
    * likely file or module
    * next action
6. Push draft to Slack for approval.
7. Create a Linear ticket if approved.

Ticket session

When a new Linear ticket arrives, the agent should:

1. Read ticket title and description.
2. Identify affected repo and service.
3. Pull codebase context.
4. Draft:
    * implementation approach
    * affected files
    * dependencies
    * risks
    * acceptance criteria
5. Push draft to Slack for approval.
6. Optionally create subtasks.

User flow

Incident flow

1. Alert enters system.
2. Session created.
3. Context gathered.
4. AI drafts incident summary.
5. Slack message sent.
6. Human approves.
7. Linear ticket created or updated.
8. Email sent if needed.

Feature request flow

1. Linear ticket created.
2. Session created.
3. Repo context gathered.
4. AI drafts implementation plan.
5. Slack message sent.
6. Human approves.
7. Ticket gets updated with plan.

⸻

Rules for the agent

Rule	Meaning
Do not modify code	Analysis only in v1
Do not deploy	No production actions
Always cite context sources internally	Repo, logs, tickets, alerts
Always produce a confidence score	Helps human review
Always send draft to Slack	Human approval gate
Never act without approval	Keeps risk low
